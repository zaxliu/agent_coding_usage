# Single-Page Console Layout Design

## Goal

将当前分裂的 `看板 / 设置` 双视图改成单页控制台，让以下信息在一个页面内形成稳定布局：

- 系统状态
- 最近任务
- 配置摘要
- 采集 / 同步 / 诊断操作
- 趋势图与明细数据

目标不是增加新能力，而是重排信息架构，让用户不需要在不同区域和不同页签之间来回寻找“当前状态”和“下一步动作”。

## Layout Direction

采用双栏单页控制台：

- 左侧为固定控制列
- 右侧为主工作区

这个结构优先保证“状态始终可见”，同时让“操作和结果”保持连续阅读流。

## Left Column

左侧固定控制列按以下顺序展示：

### 1. Brand / Runtime Identity

展示产品名和一句简短说明，弱化装饰性文案，强调这是本地控制台。

### 2. System Status

将原先分散在 sidebar 和主内容区顶部的状态合并为一组卡片，至少包含：

- 后端类型与版本
- 环境文件路径
- 最近更新时间

状态区应该能回答“系统当前能不能工作”。

### 3. Latest Job

单独展示最近任务的：

- 类型
- 当前状态
- 更新时间
- 简短摘要

状态和任务不再拆开，避免用户在不同区域拼接上下文。

### 4. Configuration Snapshot

不是完整设置表单，而是摘要型信息，用于回答“当前系统正基于什么配置运行”。

至少包含：

- 用户名
- 时区
- 回看天数
- 已配置远端数量

这里放一个明确的“编辑设置”入口，点击后在同页展开设置区，而不是跳到单独视图。

## Right Column

右侧主工作区按以下顺序展示：

### 1. Operation Bar

把高频操作集中到同一条操作区：

- 诊断
- 采集
- 同步预览
- 同步
- 保存设置 / 校验设置（如果设置面板已展开）

这组操作必须靠近结果区域，形成“执行 -> 反馈”的闭环。

### 2. Settings Panel

设置不再是独立页面，而是右侧顶部的可展开区块：

- 默认折叠
- 点击“编辑设置”或相关按钮后展开
- 展开后仍然留在当前页面上下文，不切走趋势图和任务区

设置内容仍保留现有字段，但视觉上归入单页工作流。

### 3. Dashboard Metrics

汇总卡片仍保留，但中文化并贴近主图：

- 总 Tokens
- 活跃天数
- 最高工具
- 最高模型

### 4. Trend Chart

主图保留在右侧内容流核心位置。

图表要求：

- 半对数纵轴
- 显式刻度数字
- 中文图例
- 紧凑数字单位

### 5. Comparison and Table

工具/模型对比与明细表保留在主图下方，作为“看完趋势后继续下钻”的区域。

## Interaction Rules

### Settings Visibility

- 默认折叠，减少首屏噪音
- 用户点击“编辑设置”后展开
- 保存或校验后不自动切页
- 可以保留展开状态直到用户主动收起

### Action Grouping

- 诊断、采集、同步预览、同步必须位于同一操作组
- 设置相关按钮只在设置区展开时出现，避免常态下和采集动作竞争注意力

### Status Persistence

左侧状态列在页面交互中始终可见，不受设置展开、表格过滤、弹窗输入等行为影响。

## Visual Direction

界面改成更明确的“控制台”而不是“营销式 dashboard”：

- 左侧使用更克制、更高对比的信息卡
- 右侧使用层次明确的操作条和内容区块
- 降低不必要的 hero 感
- 强化信息块边界和标题层次

不追求视觉夸张，重点是扫描效率和分组清晰。

## Testing Impact

实现后至少要验证：

- 页面不再依赖 `看板 / 设置` 切页
- 设置区可在单页内展开
- 状态区与最近任务仍然正确刷新
- 操作按钮仍能触发原有接口
- 现有 modal 输入流程不受布局调整影响

## Scope Boundaries

本次只做信息架构和页面布局重排，不做：

- 新后端接口
- 新业务动作
- 设置字段增删
- 图表数据结构变更

## Acceptance Criteria

- 用户进入页面后，不需要切页即可看到状态、操作和结果
- 设置入口与系统状态聚合在同一认知区域
- 采集 / 同步 / 诊断操作集中在一个明确操作区
- 最近任务和系统状态不再分散在多个区域
- 页面在桌面和移动端都能保持清晰分组
